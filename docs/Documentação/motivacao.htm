Motivação

Durante os últimos anos, venho desenvolvendo programas para resolver tarefas bem simples e específicas: árvores AVL, roteamento de ambulâncias, ordenação utilizando QuickSort. Em pouquissimos eu utilizei um processo de engenharia de software bem elaborado, o melhor que fiz foi documentar bem os últimos programas que eu fiz, além de adotar algumas convenções sobre a escrita do código. Desnecessária dizer que isto é insuficiente para projetos maiores.

Com meu projeto de iniciação científica, tenho meu primeiro grande software a ser construído. E este tem muitos problemas típicos do mundo real: os requisitos não são claros, não temos ferramentas para apoiar, o ambiente de trabalho não é propício para a vida humana, existem prazos, cobranças, obrigações. Para piorar as coisas, tudo está atrasado. Sem a aplicação de um processo de engenharia de software adequado, será impossível concluir este programa. Por isso, adotei um processo parecido com o Unified Process, só que este é centrado em visões.

Uma visão nada mais é do que uma opinião de algo ou alguém sobre determinada coisa. Esse opinante é conhecido, sabemos várias características sobre seu caráter, habilidades, o papel que ele desempenha no software. Agregando-os ao método, utilizando ferramentas que facilitem a sua criação e controle e, finalmente, desenvolvendo um processo que o utilize, torna-se possível a criação de um novo paradigma de engenharia de software, baseado nas observações feitas sobre o desenvolvimento de software livre ou de código aberto.
